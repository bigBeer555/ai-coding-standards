# Scope Protection Rules

本文件是最高优先级规则，优先级高于编码风格、重构建议和性能优化建议。任何任务都必须加载。

## 核心原则

- 本次修改仅允许影响当前需求相关代码。
- 禁止为了修复一个问题而修改无关逻辑。
- 禁止为了代码整洁而修改无关功能。
- 禁止为了重构而修改无关模块。
- 修复问题不等于重构项目，修复 Bug 不等于优化代码，解决需求不等于清理代码。
- 与当前需求无关的代码视为受保护代码，未经用户明确授权禁止修改、删除、重构。

## 修改前分析

开始修改代码前必须先分析：

- 当前问题真正原因是什么。
- 需要修改哪些文件。
- 为什么修改这些文件。
- 是否存在影响范围。
- 是否存在多个实现方案。

## 禁止行为

- 未分析直接修改代码。
- 未确认方案直接开始实现。
- 为“顺手优化”扩大修改范围。
- 在未确认前删除或重构代码。

## 最小修改原则

- 优先修改 1 处代码解决问题，而不是重构整个模块。
- 优先局部修复，而不是全局调整。
- 只解决用户当前提出的问题。
- 只修改解决问题必须修改的代码。
- 只删除与当前问题直接相关且已获得确认的代码。

## 删除代码保护

删除任何代码前必须确认是否与当前问题直接相关。如果无关，即使当前未调用、看起来无用、存在重复，也禁止删除。

AI 必须假设任何现有代码都可能承载：

- 新增功能。
- 灰度功能。
- 未来功能。
- 埋点统计。
- 权限控制。
- 特殊业务逻辑。

在未确认用途前禁止删除。

## 删除与重构确认机制

涉及以下行为时必须先分析并等待用户明确确认：

- 删除代码、逻辑、组件、配置、接口、状态、字段。
- 删除兼容逻辑。
- 大规模重构。

分析内容必须包含：

- 当前用途。
- 调用链。
- 影响范围。
- 是否与当前问题相关。
- 当前引用次数。
- 是否存在副作用。
- 风险等级：低 / 中 / 高。

未经用户明确同意禁止执行。

## 修改范围失控保护

当发现需要修改范围超出初始预估时，必须停止继续修改并告知用户：

- 预计新增修改文件。
- 扩大范围的原因。
- 继续执行的风险。

未经确认禁止继续扩大范围。

## 问题连续处理

用户提出的问题目标优先级高于当前执行动作。AI 必须始终关注最终要解决什么问题，而不是只关注当前执行什么操作。

当用户要求还原、回退、撤销、恢复时，默认理解为撤销当前方案，而不是放弃当前任务。回滚后仍需继续分析原始问题，除非用户明确表示不用改、先这样、算了、不处理了、结束。

出现以下情况时禁止输出“已完成”“任务结束”“修改完成”：

- 问题仍然存在。
- 报错仍然存在。
- 功能仍然异常。

## 多方案失败处理

当前方案失败时，允许按以下方式继续：

```txt
方案A -> 失败 -> 仅回滚本次失败调整 -> 方案B -> 失败 -> 仅回滚本次失败调整 -> 方案C
```

禁止方案失败并回滚后直接结束。回滚的是方案，不是需求；撤销的是修改，不是目标。
